# 海外热点聚合观察台一期任务文档

## 0. 文档定位

本文用于把 [一期需求分析文档](phase-1-requirement-analysis.md) 拆成可执行任务。它不是详细技术设计，也不是最终接口契约；后续开发时如果需求和本文冲突，以一期需求分析文档为准。

一期任务目标是完成“可配置数据源 + 每日热点榜单 + 趋势图表 + 中文摘要 + 收藏归档”的最小可用闭环。

## 1. 总体任务顺序

```text
系统骨架
  ↓
数据模型和迁移
  ↓
首批数据源验证
  ↓
采集任务闭环
  ↓
热点列表和详情
  ↓
数据洞察图表
  ↓
中文摘要和主题分类
  ↓
收藏归档
  ↓
数据源管理和任务日志
  ↓
一期验收
```

## 2. 当前建议优先级

第一步建议先做系统骨架、数据模型和 3 个稳定数据源的最小采集闭环。

首批数据源建议：

- YouTube：用于验证视频热点、地区和分类能力。
- Hacker News：用于验证开放接口和技术社区热点。
- GitHub：用于验证项目趋势和 Star 指标。

原因：

- 三个数据源官方入口相对清晰，适合快速跑通采集、去重、评分和展示。
- 可以覆盖视频、帖子、项目三种不同内容类型。
- 不依赖 X、TikTok 等高权限或高成本数据源，降低一期不确定性。

## 3. 阶段 1：系统骨架

### 目标

让海外热点聚合观察台作为独立业务系统接入 vcoding 平台，能复用统一登录和平台基础架构。

### 后端任务

- [ ] 新增 `backend/vcoding-global-trend-monitor` Maven 子模块。
- [ ] 将新模块加入 `backend/pom.xml`。
- [ ] 建立基础包结构：`api`、`application`、`domain`、`infrastructure`、`config`。
- [ ] 接入统一用户上下文和 Gateway 内部用户头校验。
- [ ] 新增健康检查或基础占位接口，验证模块可被网关访问。

### 前端任务

- [ ] 新增 `frontend/apps/global-trend-monitor-web` 应用。
- [ ] 接入 Vue Router、Pinia、Axios、Element Plus 和平台通用 UI 样式。
- [ ] 接入统一登录跳转和登录态校验。
- [ ] 建立基础布局：顶部导航、侧边导航、内容区、用户状态。
- [ ] 新增占位首页，验证登录后可进入系统。

### 验收标准

- 未登录访问系统会跳转到 `auth-web`。
- 登录成功后可以回到热点系统原目标地址。
- 前后端模块能独立启动。
- 页面风格符合平台视觉规范。

## 4. 阶段 2：数据模型和数据库迁移

### 目标

建立一期核心数据结构，为采集、展示、图表和归档提供稳定存储。

### 后端任务

- [ ] 新增数据源表 `gtm_source`。
- [ ] 新增热点内容表 `gtm_trend_item`。
- [ ] 新增指标快照表 `gtm_metric_snapshot`。
- [ ] 新增日聚合统计表 `gtm_trend_daily_aggregate`。
- [ ] 新增用户标注表 `gtm_user_trend_action`。
- [ ] 新增采集任务表 `gtm_collect_job`。
- [ ] 为平台、发布时间、采集时间、主题、内容类型、热度分建立必要索引。
- [ ] 使用 Flyway 维护迁移脚本。

### 验收标准

- Maven validate 可以通过。
- 数据库迁移脚本可重复在新库执行。
- 核心表字段覆盖需求文档中的数据对象。
- API Key、Token 等敏感字段不以明文散落在业务日志中。

## 5. 阶段 3：首批数据源验证

### 目标

在正式实现 connector 前，先验证首批数据源的字段、限流、地区、分类和异常行为。

### 任务

- [ ] 编写 YouTube 最小拉取验证脚本或临时服务方法。
- [ ] 验证 YouTube 热门视频字段：标题、频道、发布时间、播放数、点赞数、评论数、分类、地区。
- [ ] 编写 Hacker News 最小拉取验证脚本或临时服务方法。
- [ ] 验证 Hacker News 字段：标题、作者、分数、评论数、发布时间、原文链接。
- [ ] 编写 GitHub 最小拉取验证脚本或临时服务方法。
- [ ] 验证 GitHub Search 字段：仓库名、描述、语言、Star、Fork、创建时间、更新时间。
- [ ] 记录每个平台的限流、认证、异常响应和字段缺失情况。

### 验收标准

- 每个首批数据源至少能拉取一批真实数据。
- 明确每个平台可用字段和不可用字段。
- 明确每个平台的一期采集参数，例如地区、分类、关键词或语言。
- 输出数据源验证结论，作为 connector 开发依据。

## 6. 阶段 4：采集任务闭环

### 目标

完成从数据源配置到采集、去重、评分、入库和任务日志的后端闭环。

### 后端任务

- [ ] 定义统一 connector 接口。
- [ ] 定义统一中间对象 `TrendItemDraft`。
- [ ] 实现 YouTube connector。
- [ ] 实现 Hacker News connector。
- [ ] 实现 GitHub connector。
- [ ] 实现 URL、平台 ID、标题相似度的基础去重。
- [ ] 实现原始指标保存和指标快照保存。
- [ ] 实现综合热度分计算。
- [ ] 实现采集任务创建、运行、成功、失败和重试记录。
- [ ] 支持手动触发单个数据源采集。
- [ ] 支持每日定时采集。

### 验收标准

- 手动触发采集后，热点内容、指标快照和任务日志都会落库。
- 同一平台同一内容不会重复插入。
- 采集失败不会影响其他数据源。
- 任务日志能看到拉取数量、入库数量、更新数量、去重数量和失败原因。

## 7. 阶段 5：热点列表和详情

### 目标

让用户可以查看每日热点、筛选热点，并进入详情查看原始指标和历史趋势。

### 后端任务

- [ ] 实现 `GET /api/global-trend/trends`。
- [ ] 实现 `GET /api/global-trend/trends/{id}`。
- [ ] 实现 `GET /api/global-trend/trends/{id}/snapshots`。
- [ ] 实现 `GET /api/global-trend/trends/search`。
- [ ] 支持日期、平台、主题、地区、语言、内容类型、收藏状态筛选。
- [ ] 支持综合热度、平台原始热度、发布时间、采集时间排序。

### 前端任务

- [ ] 实现今日热点页。
- [ ] 实现平台榜单页。
- [ ] 实现热点详情页。
- [ ] 实现筛选栏和排序切换。
- [ ] 实现热点列表的收藏、忽略、打开原文快捷操作。
- [ ] 实现空数据、加载中、请求失败状态。

### 验收标准

- 用户可以按日期、平台、主题筛选热点。
- 用户可以查看热点详情和原文链接。
- 详情页展示原始指标、综合热度分、采集时间和指标快照。
- 页面在无数据和接口失败时有明确提示。

## 8. 阶段 6：数据洞察和图表比对

### 目标

用图表和对比视图表达趋势，帮助用户快速判断哪些平台、主题和内容正在升温。

### 后端任务

- [ ] 实现日聚合统计生成任务。
- [ ] 实现 `GET /api/global-trend/analytics/overview`。
- [ ] 实现 `GET /api/global-trend/analytics/trend-lines`。
- [ ] 实现 `GET /api/global-trend/analytics/platform-comparison`。
- [ ] 实现 `GET /api/global-trend/analytics/topic-distribution`。
- [ ] 实现 `GET /api/global-trend/analytics/ranking-changes`。
- [ ] 实现 `GET /api/global-trend/analytics/heatmap`。
- [ ] 统一图表接口的数据口径和时间窗口。

### 前端任务

- [ ] 选型并封装图表组件，优先评估 Apache ECharts。
- [ ] 在今日热点页增加概览图表区。
- [ ] 实现数据洞察页。
- [ ] 实现总热度趋势折线图。
- [ ] 实现平台对比柱状图。
- [ ] 实现主题分布图。
- [ ] 实现 Top 热点升降榜。
- [ ] 实现主题热力图。
- [ ] 实现图表点击联动热点列表或详情。
- [ ] 实现图表空数据和数据更新时间展示。

### 验收标准

- 用户可以查看今日、昨日、近 7 天、近 30 天趋势。
- 用户可以比较不同平台的热点数量、平均热度、最高热度和增长率。
- 用户可以查看主题分布和主题热度变化。
- 单条热点可以看到指标快照趋势。
- 图表不直接混加播放数、点赞数、Star 数等不可比原始指标。
- 图表视觉符合平台开发者工具风格。

## 9. 阶段 7：中文摘要和主题分类

### 目标

降低海外内容阅读成本，并为主题筛选、趋势分析和选题转化提供基础。

### 后端任务

- [ ] 定义摘要生成服务接口。
- [ ] 接入现有或新增的大模型调用能力。
- [ ] 实现标题翻译。
- [ ] 实现 100 到 200 字中文摘要。
- [ ] 实现关键词提取。
- [ ] 实现主题分类。
- [ ] 实现情绪倾向识别。
- [ ] 记录生成时间、模型来源和失败原因。
- [ ] 支持批量补摘要任务。

### 前端任务

- [ ] 在热点列表展示中文标题、摘要和关键词。
- [ ] 在详情页展示摘要、主题、情绪倾向和模型来源。
- [ ] 支持摘要缺失时手动触发生成。
- [ ] 展示生成中、生成失败和重试状态。

### 验收标准

- 已采集热点可以生成中文摘要和关键词。
- 摘要失败不会影响热点原始数据查看。
- 用户能明确区分原文内容和 AI 生成摘要。

- [ ] 实现忽略。

- [ ] 实现用户备注。
- [ ] 实现用户标签。

- [ ] 确保用户操作按当前登录用户隔离。

### 前端任务

- [ ] 实现收藏归档页。
- [ ] 实现收藏、忽略、归档、备注和标签操作。
- [ ] 支持按标签、平台、日期筛选收藏内容。

## 12. 阶段 10：一期验收和修整

### 目标

完成一期闭环验收，修复阻断性问题，补齐必要文档。

### 任务

- [ ] 按一期需求分析文档逐项核对功能。
- [ ] 检查统一登录、网关鉴权和用户数据隔离。
- [ ] 检查首批 3 个数据源采集稳定性。
- [ ] 检查图表数据口径和筛选联动。
- [ ] 检查空状态、失败状态和加载状态。
- [ ] 检查 API Key、Token、原始响应和错误日志是否存在泄露风险。
- [ ] 补充数据源接入说明。
- [ ] 补充本地启动和验证说明。
- [ ] 运行结构、后端和前端基础验证命令。

### 建议验证命令

```bash
bash scripts/verify-structure.sh
mvn -f backend/pom.xml validate
pnpm -C frontend install
pnpm -C frontend build
```

### 验收标准

- 至少 3 个稳定数据源可完成采集和展示。
- 热点列表、详情、数据洞察、收藏归档、数据源配置和任务日志可用。
- 图表能展示趋势、平台对比、主题分布和排名变化。
- 未登录访问统一跳转登录页。
- 无敏感密钥泄露到前端代码、接口响应和日志。
- 文档能支撑下一轮详细技术设计和开发排期。

## 13. 暂缓任务

以下任务不进入一期开发：

- X、TikTok、Google Trends 的自动化正式接入。
- 大规模网页抓取和登录态抓取。
- 评论区全文采集和用户画像。
- 实时秒级趋势监控。
- 多租户计费和完整 RBAC。
- 自动发布或搬运内容到第三方平台。
- 复杂 AI 趋势预测。
