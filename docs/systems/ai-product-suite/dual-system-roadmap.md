# AI 双系统产品规划：Agent 工作台与内容生成平台

## 0. 文档定位

本文是 AI 产品套件的顶层路线图，只维护 Agent 工作台、AI 内容生成与分发系统、共享 AI 能力层之间的关系、阶段顺序和边界。

AI 内容平台一期的详细页面、字段、数据表、接口和验收标准不在本文维护，统一见 [AI 内容生产与分发平台一期需求计划书](../ai-content-platform/phase-1-requirement-plan.md)。竞品和差异化分析见 [AI 内容生产与分发平台竞品分析](../ai-content-platform/competitor-analysis.md)。

## 1. 背景

当前平台已经具备统一用户中心、统一登录页、网关、示例业务系统和前端 monorepo / 后端 Maven 多模块基础。下一阶段计划在平台上建设两个 AI 业务系统：

1. Agent 工作台
2. AI 文章 / 视频 / 图片生成与多平台分发系统

这两个系统都属于 AI 能力应用，但定位不同。Agent 工作台偏“AI 能力生产资料管理”，内容生成平台偏“直接产生业务结果”。为了避免一开始做成一个边界模糊的大系统，需要先明确系统边界、依赖关系和落地顺序。

## 2. 总体定位

### 2.1 Agent 工作台

Agent 工作台负责管理 AI Agent 的配置、能力、提示词、工具权限、上下文包和运行记录。

它解决的问题是：

- 如何定义一个可复用的 Agent。
- 如何管理 Agent 的系统提示词和版本。
- 如何给 Agent 配置上下文和工具权限。
- 如何记录 Agent 每次执行的输入、输出、耗时、消耗和结果。
- 如何让多个业务系统复用同一套 Agent 能力。

Agent 工作台本身不直接面向最终内容生产结果，它更像平台的 AI 能力中台。

### 2.2 AI 内容生成与分发系统

AI 内容生成与分发系统负责从选题开始，生成文章、视频脚本、图片提示词、多平台内容版本和发布包，并管理发布任务。

它解决的问题是：

- 如何从一个选题生成内容资产。
- 如何将同一份内容适配公众号、小红书、知乎、抖音、B 站等平台。
- 如何生成可复制、可下载、可交付的发布包。
- 如何记录发布任务和发布结果。
- 如何逐步接入具备官方开放能力的平台自动发布。

内容生成系统是更容易看到明确产出的业务系统。

### 2.3 共享 AI 能力层

为了避免两个系统重复实现模型调用、任务执行、用量统计和文件处理，建议抽象一个共享 AI 能力层。

建议命名：

```text
backend/vcoding-ai
frontend/packages/ai-client
```

也可以第一阶段先不单独建模块，在内容系统内部做轻量实现，等第二阶段 Agent 工作台启动时再抽出。

共享 AI 能力层负责：

- 模型配置
- AI 调用客户端
- Prompt 渲染
- 生成任务执行
- 运行日志
- Token / 用量统计
- 文件素材引用
- 错误重试和超时控制

## 3. 三者关系

推荐关系如下：

```text
AI 内容生成系统
  ↓ 调用
内置 Agent 模板 / Agent 工作台
  ↓ 调用
共享 AI 能力层
  ↓ 调用
OpenAI / 其他模型服务 / 图片生成服务 / 视频生成服务
```

第一阶段可以简化为：

```text
AI 内容生成系统
  ↓
内置 Agent 模板
  ↓
模型调用
```

第二阶段再演进为：

```text
AI 内容生成系统
  ↓
Agent 工作台
  ↓
共享 AI 能力层
```

## 4. 为什么先做内容生成系统

建议先做内容生成系统，而不是先做完整 Agent 工作台。

原因：

- 内容生成系统有明确产出：文章、视频脚本、图片提示词、平台发布包。
- 用户更容易理解价值，不会停留在配置 Agent 的抽象层。
- 内容系统可以先内置固定 Agent，避免第一阶段过度设计。
- 内容系统跑通后，真实执行过程会反向验证 Agent 工作台到底需要哪些字段和能力。
- Agent 工作台如果先做，容易陷入“能配置很多东西，但没有实际业务结果”的问题。

## 5. 推荐落地路线

### 5.1 一期：内容生成系统 + 内置 Agent 模板

目标：

先做出一个可演示、可使用的内容生产闭环。

一期能力：

- 内容系统骨架
- 选题管理
- 内置 Agent 模板
- 文章生成
- 视频脚本生成
- 图片提示词生成
- 多平台内容适配
- 内容审核
- 发布包生成
- 发布任务记录

一期不做：

- 完整 Agent 工作台
- 用户自定义 Agent
- 多 Agent 编排
- 所有平台自动发布
- 复杂模型路由
- 团队协同审批流

### 5.2 二期：Agent 工作台基础版

目标：

把一期内容系统中内置的 Agent 模板抽象成可管理、可复用、可版本化的 Agent。

二期能力：

- Agent 模板管理
- Agent 实例管理
- Prompt 版本管理
- 上下文包管理
- Agent 运行记录
- 工具权限配置
- 模型配置
- 用量统计

二期重点不是让 Agent 自动完成所有任务，而是让 Agent 从“写死在内容系统里”变成“可以配置和复用的能力”。

### 5.3 三期：内容系统接入 Agent 工作台

目标：

内容系统不再只使用内置 Agent，而是可以选择 Agent 工作台中的 Agent 执行内容任务。

三期能力：

- 内容生成任务选择 Agent
- 平台适配任务选择 Agent
- 发布检查任务选择 Agent
- Agent 运行结果回写内容系统
- 多 Agent 串行或并行执行
- 失败重试
- 人工审核后继续执行

### 5.4 四期：平台自动发布与数据回收

目标：

逐步接入具备官方开放 API 的平台，形成从生成到发布再到数据复盘的闭环。

四期能力：

- 微信公众号草稿和发布
- 抖音内容发布
- B 站投稿
- 小红书视权限接入
- 发布状态查询
- 发布失败重试
- 阅读、点赞、收藏、评论等数据回收
- AI 复盘报告

## 7. 一期内容系统规划索引

一期先做内容生成系统，并在内容系统内部内置固定 Agent 模板，不单独建设 Agent 工作台页面。

详细需求统一维护在：

- [AI 内容生产与分发平台一期需求计划书](../ai-content-platform/phase-1-requirement-plan.md)
- [AI 内容生产与分发平台竞品分析](../ai-content-platform/competitor-analysis.md)

本文只保留一期的路线级结论：

- 第一阶段先做 `content-web + vcoding-content + 选题 CRUD`。
- 第二阶段做内容草稿和内置文章生成 Agent。
- 第三阶段做平台适配和发布包。
- 第四阶段再沉淀 Agent 运行记录，为独立 Agent 工作台做准备。

## 8. 二期 Agent 工作台详细规划

### 8.1 页面模块

```text
Agent 模板
Agent 实例
Prompt 版本
上下文包
运行记录
工具权限
模型配置
用量统计
```

### 8.2 Agent 模板

Agent 模板表示某一类可复用 AI 能力。

字段建议：

- 模板名称
- 模板编码
- Agent 类型
- 适用场景
- 默认模型
- 系统提示词
- 输入参数定义
- 输出格式定义
- 可用工具
- 是否启用
- 版本号

### 8.3 Agent 实例

Agent 实例表示绑定到某个业务系统或项目的具体 Agent。

字段建议：

- 实例名称
- 关联模板
- 所属系统
- 所属项目
- 模型配置
- Prompt 版本
- 上下文包
- 工具权限
- 状态

### 8.4 Prompt 版本

Prompt 版本用于追踪 Agent 提示词变化。

字段建议：

- Agent 模板 ID
- 版本号
- 系统提示词
- 输出格式要求
- 修改说明
- 是否当前版本
- 创建人
- 创建时间

### 8.5 上下文包

上下文包是 Agent 执行时可复用的背景材料。

内容类型：

- 项目背景
- 技术栈
- 编码规范
- 接口规范
- 内容风格
- 品牌语气
- 平台规则
- 历史示例

### 8.6 运行记录

每次 Agent 执行都应该保存运行记录。

字段建议：

- Agent ID
- 任务类型
- 输入参数
- 上下文摘要
- 输出结果
- 执行状态
- 失败原因
- 模型名称
- Token 消耗
- 执行耗时
- 创建时间

### 8.7 工具权限

Agent 可以使用的工具需要显式配置。

一期工具权限可以先做枚举：

- 文本生成
- 图片生成
- 文件读取
- 文件写入
- 网络访问
- 发布平台接口
- 代码执行

高风险工具需要人工审批。

## 9. 共享 AI Core 规划

共享 AI Core 不建议第一天就做得过重，但设计上要预留。

### 9.1 核心能力

- 模型配置管理
- AI 请求封装
- Prompt 模板渲染
- 任务执行器
- 流式输出
- 失败重试
- 超时控制
- 用量统计
- 运行日志
- 内容安全检查

### 9.2 后端对象

建议对象：

- AiProvider：模型供应商
- AiModel：模型
- AiPromptTemplate：提示词模板
- AiTask：AI 任务
- AiRun：AI 运行记录
- AiUsage：用量统计

### 9.3 何时抽出

建议在以下条件满足后抽出 `vcoding-ai`：

- 内容系统至少有 3 类 AI 任务。
- 至少两个系统需要调用 AI。
- 运行记录和用量统计开始重复。
- 模型配置需要统一管理。

在这之前，可以先在 `vcoding-content` 内部实现轻量 AI 调用。

## 10. 发布能力规划

内容发布能力按风险从低到高分四层。

### 10.1 第一层：发布包

生成平台发布包：

- 标题
- 正文
- 摘要
- 标签
- 封面提示词
- 视频脚本
- 素材清单

用户手动复制到平台后台发布。

### 10.2 第二层：半自动发布

系统提供：

- 复制按钮
- 下载 Markdown
- 下载 Word / 文本
- 跳转平台后台
- 发布结果人工登记

### 10.3 第三层：官方 API 自动发布

接入具备官方开放能力的平台：

- 微信公众号草稿和发布
- 抖音发布
- B 站投稿

### 10.4 第四层：数据回收

发布后回收：

- 阅读量
- 点赞
- 收藏
- 评论
- 转发
- 播放量
- 发布失败原因

## 11. 推荐优先级

### 当前最优先

```text
content-web + vcoding-content + 选题 CRUD
```

原因：

- 是内容系统所有能力的根。
- 不依赖 AI 模型。
- 可以快速验证业务系统脚手架、鉴权、网关、前后端联调。
- 后续文章、视频、图片、平台适配都依赖选题。

### 第二优先

```text
内容草稿 + 内置文章生成 Agent
```

### 第三优先

```text
平台适配 + 发布包
```

### 第四优先

```text
Agent 运行记录
```

### 第五优先

```text
独立 Agent 工作台
```

## 12. 不建议现在做的事

当前不建议立刻做：

- 完整 Agent 工作台
- 自定义 Agent 编排
- 多模型统一调度
- 所有平台自动发布
- 视频自动剪辑
- 图片真实生成和素材库
- 团队审批流
- 数据分析看板

这些都可以做，但应该排在内容系统最小闭环之后。

## 13. 第一阶段验收标准

第一阶段完成后，应满足：

- 用户可以登录并进入内容系统。
- 未登录直接访问内容系统会跳统一登录页。
- 用户可以创建选题。
- 用户可以查看选题列表。
- 用户可以编辑选题。
- 用户可以归档或删除选题。
- 系统目录、后端模块、前端应用符合当前仓库规范。
- 网关可以代理内容系统接口。

## 14. 第二阶段验收标准

第二阶段完成后，应满足：

- 用户可以基于选题生成文章。
- 用户可以查看 AI 生成记录。
- 用户可以重新生成内容。
- 用户可以保存内容草稿。
- 系统记录生成任务状态。

## 15. 第三阶段验收标准

第三阶段完成后，应满足：

- 用户可以生成小红书、公众号、知乎、短视频脚本等平台版本。
- 用户可以编辑平台版本。
- 用户可以生成发布包。
- 用户可以创建发布任务。
- 用户可以人工登记发布结果。

## 16. 总结

最终目标是建设两个系统：

```text
Agent 工作台
AI 内容生成与分发系统
```

但落地顺序应该是：

```text
先做内容生成系统
  ↓
内置 Agent 模板跑通内容生成
  ↓
抽象 Agent 运行记录
  ↓
再做独立 Agent 工作台
  ↓
内容系统接入 Agent 工作台
```

这样既能尽快看到明确产出，也能为后续 Agent 平台化留下扩展空间。
